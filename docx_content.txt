Ideation &amp; Problem Formulation (15%) 2.1 Mathematical Problem Framing We frame CSAO recommendation as a Sequential Contextual Ranking Problem with real-time adaptation: Formal Problem Definition: At time step t, given: • Current cart state: C_t = {item_1, item_2, ..., item_n} • User context: U = {user_history, preferences, segment} • Temporal context: T = {hour, day, meal_time} • Geographic context: G = {city, zone} • Candidate add-on set: A = {addon_1, addon_2, ..., addon_m} Objective: Learn a ranking function f such that: f(addon_i | C_t, U, T, G) → score_i that maximizes: 1. Probability of acceptance: P(click | addon_i recommended) 2. Expected incremental value: E[AOV_lift] 3. Meal completion: completeness(C_t ∪ {addon_i}) Subject to constraints: • Latency: inference_time &lt; 200ms • Diversity: top_8_recommendations must span ≥3 categories • Business rules: high_margin_items ≥ 2, impulse_items ≤ 3 • Relevance threshold: score_i ≥ minimum_threshold (filter out low-confidence items) Why Sequential? The cart is not static—it evolves. Our model must: • Re-rank candidates after every cart modification • Adapt to incomplete → complete meal transitions • Recognize when meal is 'done' (stop pushing unnecessary items) Example Sequential Flow: • t=0: Empty cart → Recommend: Biryani, Burger, Pizza (popular items) • t=1: User adds Biryani → Re-rank: Raita (0.91), Salan (0.88), Kebab (0.76) now top-3 • t=2: User adds Raita → Re-rank: Gulab Jamun (0.84), Lassi (0.79), Papad (0.61) • t=3: User adds Gulab Jamun → Meal 90% complete → Lower recommendation intensity This is fundamentally different from static collaborative filtering or single-shot ranking. 2.2 Handling Specific Constraints Constraint 1: Incomplete Meal Patterns Problem: Users often have partial meals—e.g., only biryani without sides, or snacks without beverages. Traditional models don't recognize 'meal incompleteness.' Our Solution: • Meal Completeness Scorer: Rule-based + learned - Rules: has_main=1.0, has_side=0.7, has_beverage=0.6, has_dessert=0.5, has_starter=0.3 - Completeness = weighted sum / max_possible - Example: {Biryani} → completeness = 1.0/3.1 = 32% (incomplete) - Example: {Biryani, Raita, Lassi} → completeness = 2.3/3.1 = 74% (mostly complete) • SLM Reasoning Prompt Engineering: 'Current cart has [items]. This meal is [32%] complete—missing sides and beverage. What complementary items would complete this meal based on Indian dining patterns?' • Dynamic Thresholds: If meal &lt;50% complete → aggressive recommendations (show 8 items). If &gt;80% complete → gentle suggestions (show 3-4 items, lower priority). Constraint 2: Data Sparsity (Uneven User Histories) Problem: 30% of users have &lt;3 orders, making collaborative filtering unreliable. Our Solution: • Multi-level fallback hierarchy (detailed in Section 1.3) • SLM zero-shot reasoning bridges the gap—even without user history, SLM can reason about meal logic • Aggregate embeddings at multiple granularities: user → city-segment → city → global • Quick adaptation: New users get personalized after just 1-2 orders (not 10-20 like traditional CF) Constraint 3: Diversity Requirements Problem: Avoid recommending only high-margin desserts or only one category. Our Solution: • Category Diversity Enforcement: - Top-8 must include: ≥2 sides, ≥1 beverage, ≥1 dessert, ≥1 high-margin, ≥2 categories different from cart items - Implemented via constrained beam search during ranking • Margin Balance in Prompt: 'Suggest 8 add-ons: 2 high-margin items, 3 medium-margin complements, 2 low-margin popular items, 1 surprise item (different category). Prioritize meal completion over margins.' • Post-processing filter: If top-8 fails diversity check, force-inject missing categories from next-best candidates Constraint 4: Cold Start (Covered in Section 1.3) Constraint 5: Real-Time Latency (&lt;200ms) Our Solution: • Hybrid execution: - Graph candidate generation: 10-15ms (returns top-20 candidates via graph traversal) - SLM re-ranking: 100-150ms (using quantized 4-bit Phi-3.5-mini on CPU) - Feature retrieval: &lt;30ms (from Redis + ChromaDB) - Total: 140-195ms (within budget) • Fallback: If SLM inference exceeds 180ms → return graph-only rankings (still better than no recommendations) • Caching: Pre-compute SLM responses for common cart patterns (e.g., 'solo biryani cart' → cached response). Cache hit rate: ~30-40% → those requests served in &lt;50ms 3. Model Architecture &amp; The "AI Edge" (20%) 3.1 Overall System Architecture Our system is a hybrid three-stage pipeline that combines the speed of graph-based retrieval with the intelligence of language model reasoning: Stage 1: Candidate Generation (Graph-Based Retrieval) • Input: Current cart C_t, user_id, context • Process: 1. Build/update temporal session graph (NetworkX, in-memory) 2. Compute edge weights: PMI × recency_decay × context_relevance 3. Graph traversal: BFS from current cart items to find connected candidates 4. Score candidates by: path_weight + node_centrality + category_diversity • Output: Top-20 candidate items (ranked by graph score) • Latency: 10-15ms • Purpose: Fast filtering of item space (1000s of items → 20 candidates) Stage 2: Feature Enrichment • Input: Top-20 candidates from Stage 1 • Process: 1. Retrieve user embedding from ChromaDB 2. Lookup candidate item features from Redis 3. Compute cart-candidate interaction features (price compatibility, category alignment) 4. Assemble context vector: {cart_state, user_pref, temporal, geographic} • Output: Enriched candidate set with full feature vectors • Latency: 20-30ms Stage 3: SLM Re-Ranking &amp; Reasoning (The AI Edge) • Input: Top-20 enriched candidates + full context • Process: 1. Construct natural language prompt with multiple components: - System message: 'You are a meal recommendation expert for Indian cuisine.' - Current cart details: 'Current cart: [Chicken Biryani ₹250, Veg]' - User context: 'User: Mumbai, dinner time, budget-conscious, usually orders biryani with raita' - Candidate items: 'Graph suggests: [Raita, Salan, Gulab Jamun, Lassi, Kebab, Naan, Papad, ...]' - Task instruction: 'Re-rank these 20 items. For each, provide: Rank (1-20), Confidence score (0-1), Reasoning (why this item completes/enhances the meal)' - Priority guidance: 'Prioritize meal completion, then margin, then novelty. Output JSON.' 2. SLM inference via Ollama (Phi-3.5-mini, 4-bit quantized, CPU) 3. Parse JSON response → extract rankings + scores 4. Apply business rules &amp; diversity constraints (post-processing) • Output: Final ranked top-8 recommendations with explanations • Latency: 100-150ms Why This Architecture is Superior: • Speed: Graph stage eliminates 99% of candidates instantly • Intelligence: SLM provides human-like reasoning (not just correlation) • Scalability: Graph operations are O(n) in cart size, SLM sees only 20 candidates • Interpretability: SLM outputs reasoning → can debug and explain recommendations • Adaptability: Can update graph logic or SLM prompts without retraining entire system 3.2 Temporal Session Graph: Technical Deep-Dive The temporal session graph is our core innovation for handling real-time cart updates: Graph Structure: • Nodes: Each cart item becomes a node with attributes: {item_id, name, category, price, cuisine, veg_flag, spice_level, restaurant_id} • Edges: Temporal co-occurrence relationships with dynamic weights: - Edge (item_i, item_j) weight = PMI(i,j | context) × recency_factor × time_decay - PMI (Pointwise Mutual Information): P(i,j together) / [P(i) × P(j)] - Computed from historical data: 'How often do users order i and j together vs. by chance?' - Context-aware: PMI is conditioned on meal_time, city, user_segment Example PMI values: - PMI(Biryani, Raita | dinner, Mumbai) = 2.8 (very strong) - PMI(Biryani, Dosa | dinner, Mumbai) = -0.5 (negative, unlikely together) - PMI(Biryani, Gulab_Jamun | dinner, Mumbai) = 1.2 (moderate, dessert complement) • Temporal dynamics: - Recency factor: Recent co-occurrences weighted higher - recency_weight = 1.0 for orders in last 7 days, 0.7 for 7-30 days, 0.4 for 30-90 days - Time decay within session: If item was added 10 minutes ago, its influence decays Graph Operations: 1. Graph Construction (on cart initialization): - Create nodes for each cart item - Load pre-computed PMI matrix from cache - Compute edge weights between all node pairs - Complexity: O(n²) where n = cart size (typically 2-5 items → &lt;10ms) 2. Graph Update (on item add/remove): - Add: Create new node, compute edges to existing nodes (O(n) operation) - Remove: Delete node and its edges (O(n) operation) - Incremental update: &lt;5ms per modification 3. Candidate Retrieval: - For each cart node, traverse outgoing edges to find high-weight neighbors - Collect all neighbors (2-hops max) → candidate pool - Rank candidates by aggregated edge weight from all cart nodes - Complexity: O(k × d) where k = candidates considered, d = average degree - Typical: 1000 candidates → 20 top candidates in &lt;15ms Why Graph vs Traditional Collaborative Filtering? • Real-time updates: Graph structure changes instantly with cart modifications • Explainable: Can trace why an item was recommended (via edge paths) • Sparse data friendly: Graph edges can be populated from aggregate patterns, not just individual user history • Context-aware: Edge weights vary by meal_time, city, etc. Technical Implementation: • Library: NetworkX (Python, in-memory graphs) • Storage: Graph structure stored in Redis for session persistence • PMI matrix: Pre-computed daily, stored as sparse matrix (scipy.sparse) • Scalability: Each user session = independent graph (no shared state) 3.3 Phi-3.5 SLM: The Reasoning Engine Phi-3.5-mini is a 3.8 billion parameter Small Language Model (SLM) from Microsoft, specifically designed for reasoning tasks with efficient inference: Why Phi-3.5-mini vs Alternatives: Comparison with other options: • GPT-4 / Claude: Too expensive ($0.03-$0.10 per request), latency 1-3 seconds, can't run locally • Llama-3 (8B): Larger (8B params), slower inference, 4x memory footprint • Gemma-2B: Smaller but weaker reasoning, struggles with multi-step logic • Traditional ML (XGBoost, NN): Can't handle unstructured text, no zero-shot, requires extensive feature engineering Phi-3.5-mini advantages: • Size: 3.8B parameters → fits in 2.2 GB (4-bit quantized) • Speed: 60-80 tokens/sec on CPU (Ollama), 150-250 tokens/sec on GPU • Reasoning: Trained on reasoning-heavy datasets, strong at chain-of-thought • Zero-shot: Handles new items/restaurants without retraining • Cost: $0 (runs locally) vs $$$ for API-based LLMs • Latency: 100-150ms for our prompt length (50-80 tokens output) Our SLM Usage Workflow: 1. Model Evaluation &amp; Fine-Tuning Strategy (15%) 4.1 Validation Methodology We use rigorous temporal splitting to prevent data leakage and simulate real-world deployment: Train-Test Split Strategy: • Training set: Orders from Day 1 to Day 60 (80% of data) • Validation set: Orders from Day 61 to Day 70 (10% of data) • Test set: Orders from Day 71 to Day 75 (10% of data, final evaluation) • Critical: Time-based split ensures model never sees future data during training Why Temporal Split vs Random Split? • Random split causes leakage: Model might see user's order from Day 50 in training, then predict their Day 30 order in test • Real-world deployment: Model will always predict future orders, not random historical orders • Temporal split is more pessimistic but realistic (typically 5-10% lower metrics than random split) Session-Level Splitting: • Each user session (cart building process) is atomic—never split across train/test • Example: If user adds Biryani → Raita → Lassi in one session, all three steps must be in same split • This respects the sequential nature of cart updates Cross-Validation Strategy: • 5-fold temporal cross-validation on training set for hyperparameter tuning • Each fold: Train on Days 1-48, validate on Days 49-60 (rolling window) • Select best hyperparameters based on avg performance across 5 folds 4.2 Evaluation Metrics We measure both offline model quality and business-aligned metrics: Primary Metrics: 1. Precision@8: - Definition: Of the 8 recommendations shown, how many were actually added to cart? - Formula: (# items added from top-8) / 8 - Target: ≥0.35 (baseline collaborative filtering achieves ~0.25) - Business relevance: Directly maps to recommendation acceptance rate 2. Recall@8: - Definition: Of all items the user eventually added, how many were in our top-8? - Formula: (# items added from top-8) / (total items added in session) - Target: ≥0.55 - Business relevance: Measures coverage of user intent 3. NDCG@8 (Normalized Discounted Cumulative Gain): - Definition: Ranking quality metric that rewards correct items at top positions - Formula: DCG@8 / Ideal_DCG@8, where DCG = Σ (2^relevance - 1) / log2(position + 1) - Target: ≥0.42 - Business relevance: Users are more likely to see/click top-ranked items 4. AUC (Area Under ROC Curve): - Definition: Probability that a randomly chosen positive item ranks higher than random negative item - Target: ≥0.78 - Business relevance: Overall discrimination ability Custom Metrics: 5. Meal Completion Score (MCS): - Definition: How well recommendations complete an incomplete meal - Calculation: a. Identify incomplete carts (meal_completeness &lt; 60%) b. For each, check if top-8 recommendations include missing components c. MCS = avg(completeness_after_recommendations) - avg(completeness_before) - Target: ≥20% improvement (e.g., 40% complete → 60% complete) - Example: Cart has {Biryani}. Recommendations include {Raita, Lassi} → Meal goes from 32% → 74% complete - Business relevance: Unique to our 'meal intelligence' approach 6. Category Diversity Score: - Definition: How many distinct categories are covered in top-8? - Formula: # unique categories in top-8 / 8 - Target: ≥0.50 (at least 4 different categories) - Business relevance: Avoids monotonous recommendations Segment-Level Analysis: We break down metrics by key segments to identify performance gaps: • User segments: New users, occasional users, frequent users • Time segments: Breakfast, lunch, evening, dinner, late-night • Geography: Mumbai, Delhi, Bangalore, Tier-2 cities • Cart value: &lt;₹200, ₹200-500, &gt;₹500 • Meal completeness: Highly incomplete (&lt;40%), partially complete (40-70%), mostly complete (&gt;70%) Example segment report: Segment | Precision@8 | Recall@8 | MCS | Count ------------------------|-------------|----------|--------|------ New Users (Cold Start) | 0.28 | 0.45 | +18% | 8,200 Occasional Users | 0.37 | 0.58 | +23% | 15,400 Frequent Users | 0.42 | 0.64 | +25% | 6,400 Incomplete Carts (&lt;40%) | 0.41 | 0.62 | +28% | 12,800 Dinner Time | 0.39 | 0.60 | +22% | 18,900 Late Night | 0.31 | 0.51 | +19% | 3,200 This helps identify where model struggles (e.g., new users, late-night) and prioritize improvements. 4.3 Optimization Strategy Hyperparameter Tuning: Graph Component: • PMI threshold: Test [0.5, 1.0, 1.5] → Optimal: 1.0 • Recency decay rate: Test [0.1, 0.3, 0.5] → Optimal: 0.3 • Max candidates from graph: Test [10, 20, 30] → Optimal: 20 (diminishing returns beyond) SLM Component: • Temperature: Test [0.1, 0.3, 0.5] → Optimal: 0.3 (balance consistency vs diversity) • Max output tokens: Test [100, 150, 200] → Optimal: 150 • System prompt variations: Test 3 different prompt templates Post-Processing: • Diversity penalty: Test [-0.05, -0.10, -0.15] score reduction for repeat categories • Margin boost: Test [+0.05, +0.10, +0.15] for high-margin items Accuracy vs Latency Trade-offs: We profile three configurations: 1.
